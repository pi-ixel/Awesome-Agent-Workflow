# 测试规范

## 分层

1. 数据模型测试验证未知字段、严格数字、UTC 时间、长度、枚举和代码统计总和。
2. 领域测试验证状态机、不可变字段、父子关系、离线终态、重复、乱序和时间冲突。
3. API 测试验证匿名访问、批内顺序、逐条失败隔离、错误结构、请求上限、过滤、分页和稳定排序。
4. MySQL 集成测试验证迁移、外键、唯一约束、JSON 访问、行锁和并发写入。
5. 日志测试验证必要事件、追踪字段和敏感信息缺失。

## 关键用例

- 同批次先写父记录再写子记录成功，顺序反转时仅子记录失败。
- 相同记录、时间和规范化内容返回 `duplicate`。
- 相同记录和时间但内容不同返回 `TIMESTAMP_CONFLICT`。
- 更早时间返回 `stale`，且数据库状态不变。
- 非法状态转换和不可变字段变更被拒绝。
- 工作流从完成回退时必须显式提交 `completed_at: null`。
- `waiting_objects` 只有在起止 HEAD、完成时间、上传窗口和代码统计完整时才被接受，并计入有效行及待归因数。
- 固定 Diff 上传接口验证 Step 所有权、状态、上传窗口、文件上限和 SHA-256。
- 重复 PUT 相同 Diff 保持幂等；摘要不一致或上传中断时不替换已确认文件。
- 只有已确认 Patch 的 `patch_object_key` 能使 Dev run 完成并同步生成 `mock-v1` 归因。
- Patch 内容、摘要、对象键和本地存储路径不出现在日志中。
- 零有效行 Dev run 生成 `finalized_no_match`，归因率为 `null`。
- 归因列表、工作流详情和聚合接口对同一 Dev run 返回一致结果。
- 活跃/暂停快照忽略日期范围，但保留项目、用户和版本过滤。
- 趋势补齐没有数据的日或周。
- 相同排序值跨页时不重复、不遗漏。
- 写入和查询均可匿名访问，OpenAPI 不出现 `Authorization`，`/api/v1/auth/config` 不存在。
- 超过 100 条、JSON 超过 1 MiB、Diff 超过 10 MiB、未知字段和数字字符串被拒绝。

## 通过标准

- `pytest` 全部通过，核心领域模块保持高分支覆盖率。
- `ruff check .` 无错误。
- Alembic 在空 MySQL 上可升级到 head，并可完整降级后再次升级。
- 并发测试对同一 `record_id` 提交相同或相邻时间数据时，不产生重复行或非法状态回退。
- 失败测试输出必须显示业务输入的最小安全摘要，不输出请求头、Cookie、remote 或完整请求体。
