# 模块详细设计维度化门禁检查清单

本清单是辅助检查，不替代门禁结论。门禁结论以七个准入维度是否达标为准；五链检查用于支撑、解释或反证维度结论。正式《模块详细设计说明书》和独立《模块测试用例设计》必须能支撑后续开发与验证。`.context.md` 只用于证据和过程复核，不能弥补正式成果物缺失的开发或验证指导内容。

## 0. 正式成果物与写入职责

- 正式说明书按 9 个一级章节组织：需求背景/当前 AR 描述、外部依赖、对外接口、整体方案、数据库/表设计、受影响模块、模块交互设计、模块详细方案、附录/三方件约束。
- 未涉及章节写明“不涉及，原因”，没有生成大量空表。
- 第 8 章按需求点展开，不能只用一个总表替代正文。
- 每个需求点至少包含需求描述、现状约束摘要、TOBE 设计说明、类/接口/数据结构变化、处理流程、异常回退边界场景、配置开关环境差异、日志可观测性、可测试性与验收口径。
- TOBE 不生成测试用例和 AICoding 任务；测试用例必须来自独立《模块测试用例设计》，任务拆分交由后续 AICoding 流程。
- 正式说明书和测试设计不包含 ASIS 检索过程、证据编号表、TOBE 推导过程、反证记录、门禁采样、追踪矩阵等过程性内容；测试设计自身的覆盖矩阵除外。
- ASIS 阶段只写 `.context.md`，TOBE 阶段只写正式说明书和 TOBE context，测试设计阶段只写测试设计成果物，Gate 阶段不编辑正式说明书和测试设计正文。

## 0.1 维度化门禁建议

每轮门禁必须按维度输出检查结果，并给出可重试的整改建议。

| 维度 | 达标标准 | 未达标时建议 |
|---|---|---|
| 证据充分性 | 需求、ASIS 结论、证据索引足以支撑 TOBE | 回 ASIS 补证据后重试 |
| 边界清晰性 | 模块边界、职责、不承担范围、跨模块关系清楚 | 回 ASIS 或上游架构确认 |
| 决策定稿性 | 影响实现和验证的方案、范围、接口、验收口径均已定稿 | 回 TOBE 补设计后重试 |
| 契约完整性 | 新增/修改的函数、接口、DTO、配置、错误码、路径规则等契约完整 | 回 TOBE 补设计后重试 |
| 工程可执行性 | TOBE 给出工程落点、契约、流程、异常、配置和风险处理，足以后续 AICoding 拆分任务 | 回 TOBE 补设计后重试 |
| 验证闭环性 | 测试设计以最小充分验证集覆盖 P0/P1 设计目标、失败路径和触发风险，并给出输入、前置条件、断言、观察点和建议位置 | 回测试设计补用例；必要时回 TOBE 补可测试性输入或回 ASIS 补测试证据 |
| 风险处置性 | 触发风险有缓解、验证和回退 | 回 TOBE 补设计后重试或上游确认 |

只要任一适用维度未达标，门禁建议不能是 `可进入 AICoding`。若给出 `拆分范围后有限进入 AICoding`，必须在正式说明书和测试设计中明确可进入范围、隔离边界和验证方式。

## 1. 证据链支撑检查

- 需求/AR 能追溯到 `.context.md` 中的 ASIS 结论和证据编号。
- ASIS 证据编号能在证据索引中反查。
- ASIS 结论状态在 `.context.md` 中保持为事实 / 推断 / 待确认 / 阻塞相关。
- 推断、待确认、低置信度或阻塞相关结论没有被 TOBE 当成事实。
- 关键 TOBE 决策能在 `.context.md` 中反查 ASIS 证据。
- 正式说明书用用户可读的现状约束摘要承接关键 ASIS 约束，没有输出内部证据编号或检索过程。
- 纯新增场景下，不要求 ASIS 提供不存在的既有实现证据；但必须有模块边界、新增对象不存在的检索证据、相邻同类实现或不存在相邻实现时可替代的模块惯例证据。
- 已做 ASIS 倾向采样；高风险、跨模块、低置信度、规格漂移或核心链路场景已风险加抽。

## 2. 边界链支撑检查

- 模块边界来自当前工作区 `.sdd/software_architecture.md` 的直接声明。
- 本模块承担和不承担的内容清楚。
- TOBE 没有越界承担其他模块职责。
- 跨模块交互符合模块影响性分析和模块间交互设计。
- 没有引入循环依赖、跨层调用、外部模型穿透、共享状态扩散、旁路写入或隐式耦合。
- 跨边界改动已说明防腐层、DTO/模型翻译、依赖方向和禁止穿透内容。

## 3. 执行链支撑检查

- 每个关键 TOBE 决策落到工程位置。
- 纯新增场景下，TOBE 承接 ASIS 中的模块边界、相邻同类实现或模块惯例，没有脱离现状约束凭空设计。
- 正式说明书自洽可读，不需要阅读 `.context.md` 才能理解设计结论。
- 正式说明书没有出现 ASIS 检索过程、证据编号表、TOBE 推导过程、反证记录、门禁采样、追踪矩阵等过程性章节或表格。
- 上文或上游产物中已经明确给出的接口定义、方法/函数签名、入参出参、DTO/字段、JSON/序列化名称、错误码、状态值、Topic、Path、配置项和数据结构，已在正式说明书中原样或结构化等价保留。
- 被触发的包/模块结构、类/接口、业务流程、REST、Kafka/MQ、内部接口、数据库、配置/开关变化均已按需展开。
- 字段、DTO、接口、配置项、事件名、状态值、错误码、topic、path、时间单位和 JSON/序列化名称在摘要、图、表、工程落点、可测试性输入和风险章节中前后一致。
- TOBE 足以支撑后续 AICoding 流程拆分任务；Gate 不要求 TOBE 内置任务表。
- TOBE 写清具体工程落点、函数/类型/调用点、契约、异常语义、配置和完成判定所需设计信息。
- 新增或修改的函数、接口、DTO、配置、错误码、路径规则等写清名称、所在文件/包、导出性或可见性、签名/字段/类型、调用方、错误语义和兼容策略。
- 影响实现的“建议 / 可选 / 如有需要 / 推荐 / 尽量 / 是否需要 / 待定”已消除，或明确标为非阻断改进项且不影响当前 AICoding 范围。

## 4. 验证链支撑检查

- 测试设计中的覆盖矩阵能从 TOBE 决策/契约/风险追踪到测试用例和断言/观察点；ASIS 证据追踪保留在 `.context.md`。
- 测试设计采用最小充分验证集，优先覆盖 P0/P1 设计目标、关键失败路径和触发风险；关键失败路径不能只写成可选测试。
- 测试设计没有机械枚举测试类型或一设计点一用例导致用例爆炸；可合并目标已合并，P2/低价值用例没有挤占 P0/P1 清晰度。
- 测试用例写清建议位置、前置条件/Fixture/Mock、输入/触发步骤、预期结果、断言方式和自动化方式。
- 主路径、边界输入、失败路径、兼容/回归和触发风险均已覆盖，或明确说明不适用/阻塞原因。
- 测试层级选择合理：单元测试覆盖规则和边界；模块内功能集成测试覆盖同一模块/包内业务流程和多类协作；契约、兼容、数据、配置、观测和风险专项只在 TOBE 触发时补充。
- 测试设计没有替 TOBE 补设计；若自行发明 TOBE 未定义的错误码、字段语义、状态流转或阈值，验证闭环性不通过。
- 不存在必须等全部实现完成后才能验证的关键设计。

## 5. 风险链支撑检查

只检查被触发的风险专题；未触发专题要有不适用说明。

- 架构防腐化风险有防护策略和验证方式。
- 安全风险按普通/增强强度处理。
- 性能风险按普通/增强强度处理。
- 日志可观测性能串联主流程、定位错误路径，并避免海量日志和敏感信息泄露。
- 兼容、迁移、回滚策略可执行。
- 并发、事务、幂等、重试、超时、降级、补偿按需处理。

## 6. 遗留问题闭环

- 上一轮门禁阻断问题、整改清单、复检记录、正式说明书待确认问题、正式说明书阻塞项、测试设计缺口、测试设计阻塞项、ASIS 待确认问题、ASIS 阻塞项和 TOBE 部分完成/阻塞说明已全部读取。
- 每个遗留问题保留原始编号、来源轮次和来源章节，没有用新编号覆盖或掩盖原始问题编号。
- 每个遗留问题都有闭环状态：已关闭 / 转为非阻断 / 仍阻断 / 阻塞。
- 已关闭的问题能反查到正式说明书章节、测试设计章节、`.context.md` 证据/推导记录、ASIS/TOBE 更新记录或外部输入确认。
- 转为非阻断的问题说明了不影响 AICoding 准入的理由、剩余风险和后续跟踪位置。
- 仍阻断或阻塞的问题继续进入本轮阻断问题或阻塞说明，没有被静默移除。
- 存在未逐项复核的遗留问题时，本轮门禁结论不能为通过。

## 7. 阻断分类

- `EVIDENCE`：证据不足或不确定性被伪装成事实。
- `BOUNDARY`：边界、职责、交互或防腐化不成立。
- `EXECUTION`：TOBE 不足以支撑后续 AICoding 拆分、测试设计不可执行、验证不可追踪，或正式说明书丢失上文已经明确给出的接口契约、方法签名、入参出参、DTO/字段等开发必需信息。
- `RISK`：触发风险没有缓解、验证或回退。

每个阻断问题必须关联至少一个未达标维度，并给出下一轮重试通过条件。
