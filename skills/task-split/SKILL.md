# task-split

## 定位与边界

基于已确认的**详细设计文档**和**测试用例设计文档**，拆分成独立的任务文件。详细设计内容从详设文档提取，结构化用例从测试用例设计文档提取，使得开发者拿到**单个 task 文件**即可完成理解设计、开发实现、执行验证的完整闭环。任务按串行执行顺序逐个执行，同一时刻只能开发一个任务。

**本 skill 不负责**：重新做设计、完整代码探索、补充输入文档缺失内容。任务的详细设计内容全部从详设文档中提取；用例设计全部从测试用例设计文档中提取。输入文档不足以支撑的部分标 `【存疑】`，不做猜测补充。

**有限代码核验**：本 skill 不负责重新进行完整代码探索，但允许对详设文档中明确提到的模块、类、接口、配置、数据库对象和测试目录进行有限核验，用于确认：代码对象是否真实存在、文件和模块归属是否正确、设计中的代码定位是否明显过期、多个 task 是否会修改同一核心代码区域、测试文件位置是否与设计描述一致。有限核验不得扩展为重新分析整个模块或重新设计方案。发现详设文档与代码事实不一致时，应标记 `【存疑】`，不得擅自按代码现状修改已确认设计。

## 输入与前置条件

两个输入文档**均为必需**：

| 输入文档 | 默认路径 | 来源 | 用途 |
|---------|---------|------|------|
| 详细设计文档 | `.sdd/{SR}/{AR}/{AR}-{需求短名}-{模块组名}模块详细设计说明书.md` | `module-tobe-design` 产出或用户提供 | 提取背景与目标、实现方案、接口设计、代码影响、流程图、非功能约束 |
| 测试用例设计文档 | `.sdd/{SR}/{AR}/{AR}-{需求短名}-{模块组名}模块测试用例设计.md` | `module-test-design` 产出或用户提供 | 提取结构化用例及测试覆盖要求 |

skill 启动后第一步即校验：
- 用户已指定 SR 编号、AR 编号、模块组名称或文档路径，且两个文档都真实存在
- 若详细设计文档或测试用例设计文档是对应 Skill 的标准成果物，应检查其状态及未闭环问题；两份文档均不存在影响任务拆分的未闭环问题和待确认问题时，直接继续
- 任一文档存在未闭环问题或待确认问题时，提示用户优先返回对应设计阶段处理；用户明确授权后，可以基于当前版本生成任务拆分草案
- 若任一文档不是标准成果物，或没有状态字段，则由用户明确确认其可以作为任务拆分依据

若找不到任一输入文档，**立即中止拆分流程**并引导用户：
- 找不到详细设计文档 → 提示"本 skill 需要详细设计文档。请先运行 `module-tobe-design` 生成，或提供包含实现方案、接口设计、代码影响的详细设计文档。"
- 找不到测试用例设计文档 → 提示"本 skill 需要测试用例设计文档。请先运行 `module-test-design` 生成，或提供包含结构化测试场景的测试用例设计文档。"

只有当两个输入文档都满足条件时，才进入工作流程。

## 输出成果物

输出目录：`./.sdd/{SR编号}/{AR编号}/{模块组名称}_tasks/`

- `overview.md` — 任务概览（元信息、执行规则、串行执行顺序、任务概览表、依赖链、挂账用例登记、存疑汇总）
- `T1-[任务标题].md` — 独立任务文件（至少一个）
- `T2-[任务标题].md` — ...
- ...

若由 `aaw-workflow` 编排调用，完成文件生成后需按工作单的 `data_schema` 回填任务标题列表。该列表只填写任务标题本身，不包含 `T1-`/`T2-` 等编号前缀，也不包含 `.md` 后缀。例如生成了 `T1-用户CRUD.md` 和 `T2-权限校验.md`，提交给 CLI 的数据应为：

```json
{"tasks":["用户CRUD","权限校验"]}
```

## 工作哲学

> 任务拆分不是把设计切碎，而是把设计重新组织成一条可执行、可验证、低冲突、可渐进交付的实现路径。

**八条原则**：

1. **以可验证结果为任务单位**，而不是按文档章节或代码层次机械切分
2. **保持设计意图、实现修改和验证结果之间的完整追踪关系**
3. **优先形成纵向闭环**，避免产生无法独立验证的横向任务
4. **遵循最小充分原则**，任务应足够小便于理解、实现、验证和提交，同时应足够完整能够产生独立可验证结果。预计核心代码改动超过 400 行时，必须重新检查是否可以按子功能、接口边界、状态阶段或独立验证结果进一步拆分。400 行仅作为任务过大的复查信号，不作为机械拆分或合并标准
5. **验证伴随实现**，每个任务都必须说明如何判断完成和正确
6. **依据真实技术依赖安排顺序**，并尽量降低修改冲突和返工
7. **以已确认设计为主要依据，以必要的代码核验为事实校验**，不用模型推断掩盖设计缺失；无法从输入文档和必要核验中确认的信息标记 `【存疑】`
8. **把任务拆分视为一次可落地性检查**，为开发设计一条安全、渐进的交付路径

**任务编号、逻辑依赖与执行顺序**（原则 6 的操作化）：

- `任务编号`：按照最终确定的串行执行顺序依次分配，T1、T2……TN 应与实际执行顺序一致
- `前置任务`：表示真实技术依赖，不得为了表达串行执行而添加虚假依赖；所有没有真实技术依赖的根任务填写"无"，存在依赖时填写全部直接前置任务编号
- `任务依赖图`：由真实前置任务关系形成的有向无环图
- `串行执行顺序`：对任务依赖图进行拓扑排序后确定的唯一执行序列
- 当多个任务同时满足开工条件时，按任务标题稳定性、修改冲突和编号顺序确定先后，再依次编号
- 本工作流不允许并行开发

**挂账用例规则**（原则 3、5 的操作化）：

- 每个 task 必须至少包含一组在当前 task 完成后即可执行的核心验证
- 跨 task 的端到端用例，应归属到最早具备完整执行条件的后续 task
- 只有不影响当前 task 主要完成状态的补充性验证，才允许登记为挂账用例
- 挂账用例必须记录所在任务、依赖任务、无法当前执行的原因、最终责任任务和状态
- 不允许一个 task 的全部主要验收用例都挂账到后续任务

**模板规则**（原则 7 的操作化）：

- 每次生成前，先读取对应模板：`<skill-dir>/references/overview_template.md` 和 `<skill-dir>/references/task_template.md`
- 先把模板骨架完整复制，再逐项替换占位内容；不得从空白文档自由发挥
- 模板是输出骨架和约束。必须保留模板中的标题层级、章节顺序、表格列、必填章节；占位提示要替换为实际内容，不得改名、删减、重排、加编号、合并章节或自由发挥结构
- 模板中不适用的章节填写"本任务不涉及"，不得删除章节或编造内容
- 对无法从输入文档确定的信息，不得猜测。必须在对应位置标 `【存疑】`，并同步写入 `overview.md` 的 Open Questions
- 交给用户检查时，如果 `overview.md` 的"存疑汇总（Open Questions）"非空，必须主动单独列出这些存疑项和待确认问题，并明确需要用户确认后才能视为该阶段通过

## 具体流程

先建立 todo list，并按以下流程执行。

### Phase 1: 校验输入

- 确认 SR 编号、AR 编号和模块组名称
- 确认详细设计文档路径和测试用例设计文档路径
- 若用户已提供文档路径，优先从路径或文档元信息中解析上述信息；只有无法可靠解析时，才向用户确认
- 校验两个输入文档均真实存在，任一不存在则按"输入与前置条件"中的指引中止流程

### Phase 2: 拆分任务并生成独立文件

当前 Agent 根据已确认的详细设计文档和测试用例设计文档完成任务拆分。若当前环境支持子代理，且输入文档较长、任务规模较大或依赖关系复杂，可以自行决定使用子代理辅助局部分析或初始生成；但子代理不是本 Skill 的必要执行条件。无论是否使用子代理，都必须遵守相同的输入要求、拆分规则、模板规则、自检要求和输出要求。

若存在 `software_architecture.md`，用它校验模块归属；若不存在，以详设文档中的模块为准。

**生成任务文件前，先完成以下判断**：

1. 从详细设计文档中识别本次变更的主要目标、代码影响、接口变化、流程变化、异常处理和非功能约束
2. 从测试用例设计文档中识别测试场景及其验收要求
3. 按可验证结果形成候选任务，不按文档章节、代码文件或代码层次机械切分
4. 检查每个候选任务是否包含多个可以独立交付的目标，或是否小到无法独立验证
5. 分析任务之间的真实技术依赖、共享修改区域和推荐执行顺序
6. 确认详细设计中的代码影响、接口、流程、非功能约束和测试场景均有任务承接
7. 判断完成后，直接使用模板生成 `overview.md` 和各 task 文件

**拆分与分配规则**：

1. **最少一个任务**：无论设计多简单，都必须至少生成一个任务文件 `T1-[任务标题].md`；禁止只生成 `overview.md` 而不生成任务文件——改动很小时，把全部改动合并为单个 T1 任务

2. **详细设计内容分配**：
   - 从详设文档中提取"背景与目标"、"实现方案"、"接口设计"、"非功能约束"，按任务涉及的变更关注点分配到对应 task 的"详细设计"章节
   - 当前 task 开发和测试所必需的设计信息，必须完整包含在该 task 文件中，不得仅通过引用 overview.md 承载。当前 task 的实现若依赖某流程图，则必须将原图复制到 task 文件中，并说明"本任务关注节点"和"非本任务范围"。全局流程图可以同时放入 overview.md，但 task 自包含优先。当前 task 不涉及流程变化，且理解任务不依赖相关流程图时，保留流程图章节并填写"本任务不涉及流程变更"，不得机械复制无关流程图。不得自行重画、删减或简化会影响设计语义的流程图
   - 从详设文档的"代码影响"章节按变更关注点分配到各 task 的"代码影响"表格
   - 不自行补充或重新设计；输入文档不足以支撑某个 task 的部分，标 `【存疑】` 并写入 overview Open Questions

3. **用例设计生成**：
   - 从测试用例设计文档中提取测试场景，转换为结构化用例（前置条件/测试步骤/预期结果/后置条件）并分配到对应 task
   - 如果测试用例设计文档缺少某个 task 的用例/测试描述，在该 task 的"用例设计"章节标注 `【存疑】测试用例设计文档未提供该任务的用例设计，需补充`，并同步到 overview Open Questions
   - 用例格式必须结构化：前置条件、测试步骤（编号、操作、输入参数）、预期结果（每步预期、最终状态）、后置条件
   - 不写场景式叙事；在测试用例设计文档能够提供或明确推导的范围内，用例必须包含具体操作、输入参数和预期值；输入文档未给出关键参数或精确预期值时，不得自行编造，应在对应位置标记 `【存疑】`
   - 复杂业务任务通常包含多个正常、异常和边界用例，但具体数量以测试用例设计文档和实际覆盖需要为准

4. **overview.md 内容**：
   - 元信息：SR编号、AR编号、模块组名称、来源详细设计文档路径、来源测试用例设计文档路径、任务总数、生成时间
   - 执行规则：按串行执行顺序逐个执行；同一时刻只能开发一个任务，不允许并行；多个可开工任务按编号小者优先；当前 task 的核心验证已经通过，且剩余挂账用例不影响当前 task 主要完成状态时，挂账用例不阻断按照串行执行顺序进入后续任务
   - 串行执行顺序：明确给出完整拓扑排序序列（如 `T1 → T2 → T3`），编号应与执行顺序一致
   - 任务概览表：编号、任务标题、代码影响模块、前置任务、状态
   - 任务依赖链：每行一条 `前置 → 后置` 关系，如实表达 DAG；无前置任务的起点任务不必列出
   - 挂账用例登记：所有标了 `依赖任务` 的用例都登记到此表（用例编号、所在任务、依赖任务、无法当前执行的原因、最终责任任务、状态）
   - 存疑汇总（Open Questions）：汇总所有 task 中的 `【存疑】` 和输入文档未决项

**参考模板**：
- 概览模板：`<skill-dir>/references/overview_template.md`
- 任务模板：`<skill-dir>/references/task_template.md`

### Phase 3: 自检与用户确认

按"自检清单"四维度逐项自检，任一项不符合则先修正再交给用户检查。

完成后请用户检查 `tasks/` 目录下的所有文件：
|__ 若有问题：由当前 Agent 根据用户反馈修改成果物；修改后重新执行结构完整性、内部一致性和输入文档覆盖检查；必要时可以使用子代理辅助，但不强制
|__ 若反馈内容不属于任务拆分问题而是设计内容问题（如实现方案需要调整、接口定义有误），应引导用户通过工作流 CLI 返回详设阶段，在设计环节进行修改
|__ 若无问题且无待确认存疑 → 跳出流程，给用户简要总结（任务总数、串行执行顺序、是否有 Open Questions、与输入文档覆盖情况），并建议下一步运行 `task-dev`

## 自检清单

自检按四个维度逐项执行，任一项不符合则先修正再交给用户检查。

### 结构完整性

template-first 原则下骨架复制即保证格式，此处校验骨架未被破坏：

- [ ] overview.md 骨架完整：章节/表格列/`## 执行规则`原文完整保留
- [ ] overview.md 包含完整串行执行顺序，且编号与执行顺序一致
- [ ] 每个 task 文件骨架完整：章节（任务元信息/详细设计/用例设计/DoD Checklist/交付文件）、表格列完整保留；`前置任务`字段必填，所有没有真实技术依赖的根任务填写"无"
- [ ] 模板中不适用的章节已填写"本任务不涉及"，未删除章节或编造内容

### 内容合规性

- [ ] `tasks/` 目录下至少存在一个 `T1-[任务标题].md`
- [ ] 每个 task 只有一个主要目标，并能够产生独立可验证结果
- [ ] 每个 task 至少包含一组完成后即可执行的核心验证
- [ ] 不存在全部核心验证均挂账到后续任务的 task
- [ ] 用例数量由输入测试设计和任务复杂度决定，未机械凑数
- [ ] 正常、异常和边界场景在适用于当前 task 且输入文档已有定义时均已覆盖
- [ ] 测试用例设计文档未提供必要用例时，已标记 `【存疑】`，未自行补充测试内容
- [ ] 用例中的具体参数和预期值来自输入文档或可明确推导；未自行编造示例 ID、阈值、错误码或数据库状态
- [ ] 预计核心代码改动超过 400 行的 task，已检查是否可以继续拆分
- [ ] 若超过 400 行但仍保留为单个 task，已说明不可继续拆分的原因
- [ ] 未以代码量不足 400 行为由，强制合并边界清晰、结果独立的 task
- [ ] 每个 task 的 DoD 至少覆盖功能实现、测试编写、测试执行和必要文档更新
- [ ] task 文件命名格式：`T1-[任务标题].md`、`T2-[任务标题].md`...；若需向 `aaw-workflow` 回填 `tasks` 数据，列表项只包含 `[任务标题]`，不含 `T1-`/`T2-` 前缀或 `.md` 后缀

### 产出内部一致性

overview ↔ task 文件之间：

- [ ] 所有 task 的 `前置任务` 字段指向的编号都在概览表中存在，且不存在循环依赖
- [ ] 没有真实技术依赖的任务，其 `前置任务` 字段为"无"，未因串行执行而伪造依赖
- [ ] 完整串行执行顺序与所有前置任务字段一致
- [ ] 每个 task 在串行执行顺序中出现且仅出现一次

## 完成后回调

> 若不处于 aaw-workflow 编排中，请忽略此节。

本 skill 由 `aaw-workflow` 编排调用时：

1. 确认成果物已经生成、自检通过并获得用户确认
2. 返回 `aaw-workflow`
3. 执行 `aaw next --sr <SR号> --json` 查看当前工作项
4. 若当前工作项成果物存在，且本 Skill 完成条件全部满足，则执行：`aaw done --sr <SR号> <id>`
5. 若只有成果物存在，但尚未完成用户确认或仍有影响开发的存疑项，不得执行 `aaw done`
6. 不记得 SR 号时，先执行：`aaw status --json`

**完成条件**（必须同时满足）：
- 所有任务成果物已经生成
- 自检清单全部通过
- 用户已经确认任务拆分结果
- 不存在尚未确认且会影响任务实施的 Open Questions
- overview.md 与 task 文件之间的任务数量、标题、依赖、模块和用例登记保持一致

`deliverables_exist: true` 只表示成果物文件存在，不能单独作为当前阶段完成的依据。
