# ASIS 探索问题与证据清单

本清单用于把需求、上游设计和初步代码线索转化为 ASIS 探索任务。它不是要求逐项深挖的收尾检查表；应根据本次现状关注点和技术栈选择适用项，形成问题、查证范围和证据要求。

使用方式：

1. 先理解本次需求、AR、上游设计和模块边界，列出会影响 TOBE 的现状关注点。
2. 再用本清单把关注点转成具体代码问题，例如“配置如何加载”“当前失败路径如何处理”“是否已有同类实现”“测试是否表达历史兼容行为”。
3. 对每个问题建立 ASIS 探索任务，由 SubAgent 查证。
4. 查证结果必须回到代码、配置、测试、迁移或运行注册逻辑。README、注释、架构文档和 `software_architecture.md` 可以提供线索，但不能替代代码事实。
5. 如果某类检查与本次需求无关，记录不展开原因即可；不要机械产出无关结论。

## 1. 模块边界

- `.sdd/software_architecture.md`。该文件是模块边界必要输入，缺失、不可读或目标模块缺失时中断。
- 包、命名空间、模块、工作区、Gradle/Maven/npm/pnpm/yarn/go/cargo/dotnet 工程边界。仅在 `.sdd/software_architecture.md` 已明确边界后，用于分析边界内 ASIS 事实。
- 归属元数据、CODEOWNERS、README、架构文档、ADR。仅用于补充边界内事实，不替代模块边界。
- 目标模块的 import 关系和反向引用。仅用于验证边界内调用事实，不替代模块边界。
- 构建目标、部署清单、Helm/Kubernetes 配置、Dockerfile、服务描述文件。

可转化的问题：

- `.sdd/software_architecture.md` 如何声明目标模块、目录、包路径和依赖关系？
- 用户指定模块与架构声明是否冲突？
- 哪些文件确定属于模块，哪些只是外部依赖或疑似相关？
- 本次需求相关切片是否需要扩展到相邻入口、配置、测试或依赖？

## 2. 入口

- HTTP 路由、Controller、Resource、Router、Middleware。
- RPC 或 gRPC 服务、proto 文件、OpenAPI 文件、生成的客户端。
- 消息 Consumer、事件 Listener、流处理器。
- 定时任务、cron 配置、任务队列、Worker。
- CLI 命令、管理脚本、迁移脚本、批处理任务。
- 框架注册、依赖注入、插件注册、自动配置。

可转化的问题：

- 本次需求会从哪些入口触发？
- 入口到模块内部的真实调用链是什么？
- 是否存在绕过主入口的批处理、异步消费、定时任务或脚本入口？
- 框架注册或自动配置是否隐藏了运行时入口？

## 3. 内部组件

- 应用服务、领域服务、用例、命令/查询处理器。
- 领域模型、实体、值对象、聚合、状态机。
- Validator、Policy、权限校验、Guard。
- Repository、DAO、Mapper、Query Builder。
- Adapter、Client、Gateway、Facade、防腐层。
- 承载业务语义的工具类，而不是普通通用工具函数。

可转化的问题：

- 当前职责由哪些服务、模型、仓储或适配器承担？
- 是否已有同类实现、抽象或防腐层可以作为 TOBE 约束？
- 现有设计模式是稳定惯例，还是只在个别代码中偶然出现？
- 如果没有通用惯例，哪些证据能证明“未发现可复用惯例”？

## 4. 数据与状态

- 数据库 Schema、迁移、种子数据、Fixture、ORM 映射。
- 表、索引、唯一约束、可空列、外键。
- 缓存 Key、TTL、失效路径、本地缓存。
- 消息 Schema、事件名、Topic、Queue、分区 Key。
- DTO、请求/响应 Schema、序列化字段别名、枚举值。
- 文件格式、对象存储路径、路径命名规则。

可转化的问题：

- 本次需求涉及哪些表、字段、索引、缓存 Key、消息 Schema 或 DTO 字段？
- 当前状态变化、持久化、副作用和外部事件如何发生？
- 是否存在默认值、兼容别名、历史字段、哨兵值或空值约定？
- 测试 Fixture 是否揭示了代码中不明显的数据约束？

## 5. 运行时配置

- 各环境配置文件。
- 环境变量和密钥名称。
- Feature Flag、灰度开关、熔断或关闭开关。
- 超时、重试次数、批大小、分页默认值、限流配置。
- 按地域、租户、客户、账号、版本、部署模式区分的条件行为。

`software_architecture.md` 可能提到配置文件的位置和作用，但不会覆盖全部配置事实。配置结论应以代码、配置文件、配置绑定类、启动注册逻辑、测试和部署清单为准；如果文档与代码不一致，记录为规格漂移。

可转化的问题：

- 配置文件在哪里，配置项如何被绑定、读取和默认化？
- 不同环境、租户、版本或部署模式是否存在条件行为？
- 功能开关关闭后当前 ASIS 行为是什么？
- 是否存在未在架构文档中提到但会影响本次设计的配置？

## 6. 跨模块交互

- 对其他模块的直接 import。
- API、RPC、事件、数据库、缓存或文件耦合。
- 共享常量、共享 DTO、共享数据库表。
- 启动顺序、初始化顺序、调度顺序、最终一致性假设。
- 对调用方或下游消费者的向后兼容保证。

可转化的问题：

- 当前模块与相邻模块通过什么契约交互？
- 数据、接口、事件或配置的归属是否清楚？
- 是否存在共享表、共享 DTO、共享常量或跨层调用导致的边界腐化？
- 相邻模块是否需要配合修改；如果需要，ASIS 能否支撑职责边界判断？

## 7. 失败与保护行为

- 异常类、错误码、错误映射、用户可见文案。
- 重试、超时、熔断、降级、补偿。
- 事务边界、锁、乐观版本、唯一性保护。
- 幂等 Key、去重表、消息重放处理。
- 局部失败行为和清理路径。
- 安全、权限、租户隔离、数据脱敏、审计事件。

可转化的问题：

- 当前失败路径如何返回、重试、补偿或回滚？
- 是否存在幂等保护、唯一性保护、锁或事务边界？
- 权限、租户、审计和脱敏逻辑是否与本次需求有关？
- 局部失败是否会留下需要 TOBE 处理的中间状态？

## 8. 可观测性与运维

- 日志字段、Trace Span、指标名、告警名。
- 健康检查、就绪检查、看板、运行手册。
- 运维脚本、数据修复脚本、回填脚本。
- 写在脚本或文档里的已知支持流程。

可转化的问题：

- 当前日志、指标、Trace 或告警是否足以验证本次变更？
- 运维脚本、回填脚本或修复脚本是否体现隐藏运行流程？
- 是否存在只在文档或脚本中体现、代码入口不明显的支持流程？

## 9. 测试作为证据

- 覆盖业务规则和边界场景的单元测试。
- 覆盖数据库、缓存、队列、RPC、HTTP 的集成测试。
- 契约测试、Golden File、快照测试。
- 与历史缺陷或兼容行为相关的回归测试。
- 揭示必填默认值的测试 Fixture 和 Builder。
- 被跳过、不稳定、忽略或隔离的测试。

可转化的问题：

- 哪些测试直接表达当前业务规则、边界场景和兼容行为？
- 哪些测试 Fixture 或 Builder 揭示默认值、必填字段或历史约定？
- 本次需求相关行为是否存在未覆盖风险？
- 被跳过或隔离的测试是否暗示现有机制不稳定？

## 10. 隐藏约束搜索词

搜索与技术栈匹配的变体：

- `TODO`、`FIXME`、`HACK`、`XXX`、`deprecated`、`legacy`、`compat`、`backward`、`temporary`。
- `retry`、`timeout`、`lock`、`transaction`、`idempotent`、`dedupe`、`duplicate`。
- `feature`、`flag`、`gray`、`canary`、`enable`、`disable`、`kill`。
- `permission`、`auth`、`tenant`、`audit`、`mask`、`encrypt`、`decrypt`。
- `migration`、`backfill`、`repair`、`cleanup`、`rollback`。
- `null`、`default`、`fallback`、`unknown`、`other`、`none`、`empty`。
- `limit`、`max`、`threshold`、`capacity`、`size`、`path`、`archive`、`catalog`、`default`，用于发现文档与代码之间的阈值、路径、默认值规格漂移。

## 11. 证据标准

每个重要结论至少捕获以下一种证据，并分配稳定证据编号，例如 `E1`、`E2`：

- 文件路径加行号；如无法定位行号，再记录类名、函数名或方法名。
- 配置 Key 加配置文件或环境来源。
- 测试文件加测试名。
- 迁移文件加表、索引或字段。
- API 路由、事件名、Topic、Job 名或命令名。
- 分析过程中生成的简短命令输出摘要。

ASIS 阶段必须在同名前缀 `.context.md` 中记录稳定证据编号、探索任务编号和 ASIS 结论编号。后续正式说明书中的重要结论必须引用这些证据编号，并能在 `.context.md` 的证据索引中反查到具体位置和支撑结论。ASIS 阶段不得创建或编辑正式说明书。

没有直接证据但与本次设计相关的结论，必须标记为 `推断` 或 `待确认`，并在待确认问题或 ASIS 阻塞项中说明影响。

## 12. 无惯例或无同类实现时的处理

如果新仓库、目标模块或新增能力没有可复用的通用惯例，不要为了补齐 ASIS 而编造规则。应记录：

- 检索过的目录、文件类型、关键词和同类实现范围。
- 未发现同类实现或通用惯例的证据。
- 仍可确认的模块边界、工程组织、配置加载、测试组织或依赖注册事实。
- 该缺口对 TOBE 的影响，例如需要 TOBE 建立新约定、请求用户确认或拆分相邻模块设计。

无惯例本身可以成为 ASIS 结论，但必须带证据和影响说明。
