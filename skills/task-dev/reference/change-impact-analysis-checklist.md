# 软件变更影响分析开发阶段自检 Checklist

## 1. 使用说明

本 Checklist 供 `task-dev` 在代码实现、测试验证和任务交付阶段使用，承接 `sr-design` 的[《软件变更影响分析设计阶段自检 Checklist》](<skill-dir>/sr-design/reference/change-impact-analysis-checklist.md)。它以任务文件和上游设计中的变更影响分析为基线，通过实际代码差异和测试证据验证设计结论。`sr-design` 不加载或执行本文件。

结果栏填写：

- 通过：实际实现和验证满足检查要求，且证据充分；
- 不通过：存在缺失或不符合项；
- N/A：不适用，必须说明原因；
- 待确认：证据不足，必须明确阻塞影响和后续处理方式。

证据位置可填写任务文件章节、代码文件和行号、完整差异、调用链搜索结果、提交或 PR、自动化测试、日志、性能原始数据或评审结论。

执行规则：

- 每个任务均逐项判断适用性；不得只检查主要功能文件。
- 任一检查项不得只填"通过"而不提供证据。
- 实现偏离设计时，先停止并按 `task-dev` 的冲突处理规则向用户报告，不得用开发阶段结论静默改写设计。
- 存在未关闭的 P0、P1 或影响当前任务验收的"待确认"项时，不得将任务标记为 `Completed`。
- "接口签名未变"、"单元测试通过"或"代码改动很小"均不能单独证明无对外影响。
- 只有实际实现拟维持 E0 结论时才执行第 4 章快速反证；任一答案为"能"或"待确认"时，必须重新判定影响级别并回补相关设计或任务记录。

### 3.3 资源、内存和性能实现

开发阶段对所有适用的资源与性能指标均须给出详细实际数据，不得因 SR 阶段预估“影响较小”而省略。实测必须与变更前基线、SR 预估、验收门槛和相同统计口径对比。若 SR 预估较大、实测影响较大、超过门槛或明显偏离预估，必须进一步定位变化原因，分析对超时、重试、限流、容量和上层功能的影响，并给出优化、接受或回退结论。

## 4. 快速反证检查

仅在开发阶段复核分级为 E0 时，基于实际代码和验证结果逐项完成以下反证。

| 编号 | 反证问题 | 结论 | 证据 |
| --- | --- | --- | --- |
| E0-01 | 能否找到任一既有合法输入，其返回值、输出或副作用发生变化 |  |  |
| E0-02 | 能否找到任一边界或非法输入，其错误码、失败时机或输出状态发生变化 |  |  |
| E0-03 | 能否找到任一调用顺序、并发、超时或重试场景，其行为发生变化 |  |  |
| E0-04 | 能否找到任一平台、构建选项、配置或混合版本场景，其行为发生变化 |  |  |
| E0-05 | 能否找到任一上层功能依赖了本次变化的结果、时序或资源特征 |  |  |
| E0-06 | 最大输入或最大并发下，内存、CPU、延迟或容量是否发生可测变化 |  |  |
| E0-07 | 是否有新增日志、事件、指标、文件、网络请求或持久化副作用 |  |  |

若任一答案为"能"或"待确认"，不得维持 E0；应重新判定为 E1、E2、E3 或继续分析。
