# task-dev

根据 `task-split` 生成的单个任务文件完成开发和交付。每次调用只处理一个明确的 Task；`overview.md` 管理多个 Task 的顺序、依赖、状态和挂账用例，AAW Workflow/CLI 负责调度。

## 输入

```text
./.sdd/[SR编号]/[AR编号]/[模块组名称]_tasks/
├── overview.md
├── T1-[任务标题].md
├── T2-[任务标题].md
└── ...
```

必须获得当前 Task 文件；在 AAW 工作流中直接使用工作单给出的路径。当前 Task 文件应包含任务元信息、详细设计、用例设计、DoD Checklist 和总结章节。

## 单 Task 执行纪律

- **锁定当前 Task**：本轮只实现、验证和总结传入的 Task，不搜索、选择或启动其他 Task。
- **以 overview 为多 Task 事实来源**：从 overview 校验当前 Task 的状态、串行位置、前置任务和相关挂账用例；不要在 Skill 内重建调度规则。
- **前置条件不满足则停止**：当前 Task 不是 overview 中应执行的 Task、真实前置任务未完成、任务文件缺失或关键字段缺失时，不得开发。
- **相关存疑先处理**：只处理会影响当前 Task 实现的 Open Questions 或 `【存疑】`；无关问题保留在 overview，不阻塞当前 Task。
- **核心验证必须通过**：不得把当前 Task 的主要实现或全部主要验收用例挂账到后续 Task。
- **完成后停止**：完成当前 Task、更新文件并回调 AAW 后结束。本轮不进入下一 Task，即使用户要求连续开发；下一 Task 由 Workflow/CLI 在新执行上下文中调度。

## 开发流程

先建立当前 Task 的 todo list，再按以下顺序执行。

### 1. 当前 Task 开工检查

1. 完整读取当前 Task 文件和同目录 `overview.md`。
2. 确认 overview 中存在当前 Task，状态为 `Pending` 或 `In Progress`，且它是串行顺序中当前应执行的 Task。
3. 读取当前 Task 的 `前置任务` 字段；字段缺失或与 overview 冲突时停止，不按编号推断。
4. 确认所有真实前置任务均为 `Completed`，并读取它们总结中的相关变更和关键决策。
5. 检查 overview 的存疑汇总，只保留影响当前 Task 的问题；存在未解决阻塞问题时向用户确认。
6. 根据“代码影响”定位相关实现、测试、接口和直接依赖。不要在开工前完整重读架构文档、Spec 或遍历整个仓库。

### 2. 加载 AI 研发流程

检查仓库根目录 `.sdd/AICodingGuidelines.md`：

- 文件不存在：使用本 Skill 的内置开发流程。
- 文件存在：完整读取 `# AI 研发规范` 下的 `## 代码开发流程` 并执行；缺少这两个标题或流程内容不可执行时停止并报告。
- 自定义流程可以增加或调整工程步骤，但不能取消“实现设计”“实现后测试”“核心测试通过”“修改后重新验证”和本 Skill 的交付规则。

参考模板见 [references/AICodingGuidelines.md](references/AICodingGuidelines.md)。

### 3. 实现详细设计

1. 完整落实当前 Task 的背景目标、代码影响、流程图、实现方案、接口设计、错误处理和非功能约束。
2. 对照相关代码现状实现，不偏离设计中确定的上下游调用关系。
3. 遵循项目现有代码风格和命名约定，不提前实现后续 Task，不做无关重构。
4. 若设计与代码事实冲突且无法按设计实现，暂停并向用户报告；不要自行改写业务设计。

### 4. 用例实现与正确性验证

任务文件中的用例是规格，不是可执行测试。对每个用例：

1. 将前置条件、测试步骤、预期结果和后置条件落实为自动化测试。
2. 项目无测试框架或用例不适合自动化时，编写可执行的手动验证清单。
3. 若用例声明依赖其他 Task，先检查依赖状态。只有补充性验证且明确记录依赖、当前无法执行原因和最终责任 Task 时才可挂账；核心验证不得挂账。
4. 执行当前 Task 的测试，对照预期结果逐项检查。
5. 测试失败时优先修复实现。只有确认测试代码或用例描述错误后，才能修正测试或用例并重新执行；不得修改测试迁就错误实现。
6. 执行后置条件，清理测试数据并恢复状态。
7. 重复修复和验证，直到所有非挂账核心用例通过。

### 5. 约束自审与优化

正确性验证通过后，再针对本次实际变更定向检查：

1. 从 `.sdd/software_architecture.md` 读取当前模块、直接依赖和本次变更涉及的边界，检查职责和依赖方向。
2. 从 `.sdd/spec.md` 读取与本次语言、代码类型和测试有关的条款；文件不存在时跳过并记录。
3. 执行 `AICodingGuidelines.md`、仓库脚本或项目说明要求的 CodeCheck、lint、静态分析、类型检查和构建命令。
4. 审查正确性、可读性、职责划分、重复代码、错误处理、日志/指标/追踪和可维护性。
5. 仅在任务存在明确性能目标或实现具有明显性能问题时优化性能，不做无依据的提前优化。
6. 发现架构、Spec 或检查问题时修复当前实现；若修复会改变已定设计，暂停并报告设计冲突。

### 6. 修改后重新验证

自审、优化或检查修复改变代码后，必须重新执行：

- 当前 Task 的所有非挂账测试或手动验证。
- 受影响的 CodeCheck、lint、静态分析和类型检查。
- 研发流程要求的构建或专项检查。

任何核心测试或必需检查失败时，不得完成 Task。

### 7. DoD 验证

逐项执行当前 Task 的 DoD Checklist：

- **功能项**：实现符合全部验收点和详细设计。
- **测试项**：所有非挂账用例均已落成测试或手动清单并验证通过；挂账条目满足限制条件。
- **文档项**：需要更新的文档已完成。
- **可观测性项**：关键路径具备必要日志、指标或追踪。
- **工程项**：研发流程要求的检查、构建和重新验证均已通过。

仅勾选实际通过的条目。非挂账项未通过时不得将 Task 标记为 `Completed`。

### 8. 总结和 overview 更新

在当前 Task 的 `## 总结` 中写入：

```markdown
## 总结

### 变更摘要
- **修改文件**：[新增和修改文件]
- **核心实现**：[核心逻辑]
- **关键决策**：[偏离设计时说明原因；无则写无]

### 用例验证结果
- TC-T[N]-01：✅ 通过
- TC-T[N]-0X：⏸ 挂账 - 依赖 T[Y]，原因：[原因]，最终责任：T[Z]

### 工程检查结果
- [测试、CodeCheck、构建或专项检查及结果]

### 遇到的问题
- [问题和解决方式；无则写无]

### 对后续任务的建议
[仅记录后续 Task 需要消费的信息；无则省略]
```

同时更新：

- 当前 Task 的 DoD 勾选状态。
- overview 中当前 Task 状态为 `Completed`。
- 符合条件的挂账用例，沿用 overview 的六列格式：用例编号、所在任务、依赖任务、无法当前执行的原因、最终责任任务、状态。
- overview 中与当前 Task 有关且已可回归的挂账用例状态。

只报告下一 Task 信息，不读取或执行下一 Task。

## 完成检查

结束前确认：

- 本轮只处理了工作单指定的当前 Task。
- overview 已证明当前 Task 可开工，所有真实前置任务已完成。
- 不存在影响当前实现的未解决存疑。
- 详细设计、流程、接口、错误处理和非功能约束均已落实。
- 所有非挂账用例已落成测试或手动清单并通过。
- 架构和 Spec 已按实际变更定向自审。
- 必需的 CodeCheck、静态检查、构建和专项检查已通过。
- 所有后续代码修改均已重新验证。
- DoD、总结和 overview 已更新。
- 没有提前实现或启动其他 Task。

## 输出规则

- 代码遵循项目现有风格和 `.sdd/spec.md` 中相关规范。
- 总结写回当前 Task 的 `## 总结`，不另造过程文档。
- 用户只要求预览时，不更新文件、不标记 `Completed`、不执行 AAW 完成回调。
- 最终回复聚焦当前 Task 的实现、验证、挂账和风险，不自动开始下一 Task。

## 模板契约

- 当前 Task 遵循 `task-split/references/task_template.md`。
- overview 遵循 `task-split/references/overview_template.md`。
- DoD 勾选使用 `- [ ]` → `- [x]`。
- 挂账登记必须使用 overview 规定的六列格式。

## AAW 完成回调

> 不处于 `aaw-workflow` 编排中时忽略本节。

1. 确认当前 Task 的实现、核心验证、重新验证、DoD、总结和 overview 更新全部完成。
2. 使用工作单给出的完成数据文件，写入 `task_id`、研发流程来源、实现状态、测试状态、自审优化状态、重新验证状态和工程检查摘要。
3. 仅在完成条件全部满足时执行工作单的 `commands.done`。
4. `deliverables_exist: true` 只表示文件存在，不能作为完成依据。
5. 回调完成后停止，不执行 `aaw next`，不读取或开发下一 Task。
